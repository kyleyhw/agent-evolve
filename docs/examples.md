# Example runs

Two artifacts under [`examples/`](../examples/) give you a concrete feel
for what an evolution run looks like before you point Claude at a real
codebase:

| File | What it is |
|---|---|
| [`examples/agent-evolve.yaml`](../examples/agent-evolve.yaml) | A fully commented manifest — every optional field populated, suitable as a template for your own projects. |
| [`examples/demo_run.py`](../examples/demo_run.py) | An end-to-end run of the real pipeline against a toy target. No LLM calls — the three agent roles are played by hardcoded candidate variants, so you can run it offline and in CI. |
| [`examples/evolve-graph.mmd`](../examples/evolve-graph.mmd) | Static Mermaid snapshot of a 4-candidate / 3-round run (the one shown in the README). |
| [`examples/evolve-report.html`](../examples/evolve-report.html) | Hand-crafted interactive D3 report with the same 4 candidates. |
| [`examples/demo-report.html`](../examples/demo-report.html) | D3 report regenerated by the last `demo_run.py` invocation. |

This document walks through the demo run in detail, then shows how to
modify it to target a different function.

---

## The demo: optimising `fib(n)`

### What it does

`examples/demo_run.py` creates a temp working directory with two files:

```
<tempdir>/
    bench.py   ← benchmark harness (never modified)
    fib.py     ← candidate implementation (overwritten per round)
```

Then it runs four candidates across three rounds. Each "explorer" is
replaced by a hardcoded Python source string — we are not calling Claude
here, we are driving the real Python pipeline with predetermined
candidates so the run is deterministic, fast (~3 seconds), and offline.

The four variants are:

| Variant | Round | Operator | What it is |
|---|---|---|---|
| `baseline` | 1 | explore | Naive `O(2^n)` recursive Fibonacci — reference implementation. |
| `memoised` | 2 | mutate | Baseline + `@lru_cache(maxsize=None)` — trades memory for no recomputation. |
| `buggy_loop` | 2 | mutate | Forward loop with an off-by-one (`range(n-1)` instead of `range(n)`). |
| `iterative` | 3 | crossover | Correct forward iteration; conceptually combines memoised's "never recompute" with buggy_loop's "no recursion". |

For each candidate the demo:

1. Writes the variant's source into `fib.py`
2. Calls `scope.enforce_scope(["fib.py"], spec.scope)` — confirms the
   change stays within `target_files`
3. Runs `eval.run_eval(spec.eval_command, cwd=<tempdir>)` — a subprocess
   that imports `fib`, warms it up, times 100 calls of `fib(22)`, and
   prints a JSON metrics line
4. Calls `equivalence.check_equivalence(baseline_fn, variant_fn,
   tuples(integers(0..20)), samples=40)` — runs the `hypothesis`
   property-based equivalence checker
5. Calls `backend.score_candidate(id, metrics, equivalence=report)`
6. Runs a mock reviewer that reads the metrics + equivalence report and
   emits APPROVE / REQUEST_CHANGES / REJECT per a hardcoded checklist
7. Calls `backend.record_verdict(id, verdict)`

After all four candidates, the supervisor role:

8. Calls `viz.build_graph` + `viz.render_mermaid` + `viz.render_html`
9. Picks the winner (fastest APPROVE) and prunes the rest
10. Calls `backend.finalize(winner_id)` — produces `final_pr.json`
    describing the PR that *would* be opened against `main`. Nothing is
    merged; with the local backend the PR is just a JSON descriptor.

### What each candidate demonstrates

**#1 baseline — REQUEST_CHANGES.** The reviewer expects a *speedup vs the
baseline*; here the candidate *is* the baseline, so the speedup metric is
zero. This is the expected state of the very first candidate in a run —
it seeds the Trait Matrix and gives the supervisor something to mutate in
round 2.

**#2 memoised — APPROVE, winner.** Adding `@lru_cache` turns `fib(n)`
from `O(2^n)` to `O(n)` with ~25,000× measured speedup. The equivalence
checker confirms outputs match on 40 random `n ∈ [0, 20]` inputs. **But
watch out** — the benchmark calls `fib(22)` 100 times *in a loop without
resetting the cache*, so iterations 2–100 are free cache hits. The
reported 0.1 µs is "amortised over 100 calls assuming a persistent cache",
not "cost of a fresh lookup". A benchmark that resets state between calls
would give a fairer picture; this is a deliberate demonstration of how
a micro-benchmark can reward an implementation for the wrong reason.

**#3 buggy_loop — REJECT.** Off-by-one: `for _ in range(n - 1)` iterates
one fewer step than it should, so `fib(0)` returns `1` instead of `0`.
The test suite catches it (`test_pass_rate` drops to 0.0), *and* the
equivalence checker independently catches it — the counterexample
`(0,)` is reported in the verdict. Two independent rejection paths; the
reviewer rejects on either. This is the core safety story: candidates
cannot pass by getting lucky on the test suite alone.

**#4 iterative — APPROVE, not winner.** Correct `O(n)` forward loop with
no cache. Equivalent to baseline. In the hardcoded benchmark it's slightly
slower than #2 on paper (0.58 µs vs 0.10 µs), so the winner-selector
picks #2. Honest caveat — in a realistic workload with cache resets, #4
would likely beat #2, because the persistent cache is a quirk of the
benchmark, not of the implementation.

### Artifacts

**Per-candidate JSON.** Each candidate's full `EVOLVE_STATE` lives under
`<tempdir>/evolve-state/1/candidates/<id>.json`:

```json
{
  "problem_id": "1",
  "candidate_id": "2",
  "operator": "mutate",
  "round": 2,
  "status": "approved",
  "parent_ids": ["1"],
  "metrics": {"duration_us": 0.1, "test_pass_rate": 1.0},
  "hypothesis": "Add @lru_cache — trades a tiny amount of memory for never recomputing the same fib(k) twice.",
  "equivalence_report": {"equivalent": true, "samples_tested": 40, "counterexample": null, "mismatch": null, "error": null},
  "reviewer_verdict": {"verdict": "APPROVE", "reason": "0.1µs vs baseline 2598.0µs.", "checklist": {...}, "confidence": "high"}
}
```

**`final_pr.json`.** The descriptor for the PR that would be opened
against `main`:

```json
{
  "kind": "final_pr",
  "winner_id": "2",
  "winner_branch": "evolve/1/candidate-2",
  "target_branch": "main",
  "reviewers": ["kyleyhw"],
  "summary": "...",
  "trait_matrix": [...],
  "merged": false
}
```

With the local backend this is just a JSON file — nothing actually opens
a PR. Switch to the GitHub backend and `finalize()` opens a real PR with
this descriptor's content in the body.

**`examples/demo-report.html`.** Interactive D3 tree. Open in a browser:

- **Drag** any node to reposition it. **Scroll** to zoom.
- **Click** a node to expand its full EVOLVE_STATE in the right panel —
  hypothesis, conclusion, metrics, reviewer checklist, verdict reason.
- **Lineage / Timeline** toggle — the default layout draws parent→child
  edges; Timeline re-lays out by round so you can see the rounds
  stratified left to right.
- **Export PNG** — renders the current layout to an image for slides or
  READMEs.
- Node colours match the Mermaid convention: green winner, blue active /
  approved, grey pruned, red rejected, tan pending.

### Reading the report for the demo run

Open [`examples/demo-report.html`](../examples/demo-report.html) after
running the demo. You should see:

- A grey pruned node for #1 (baseline, REQUEST_CHANGES → demoted)
- A blue APPROVE node for #2 sized up as the green WINNER
- A red REJECT node for #3 (the buggy loop)
- A blue APPROVE node for #4 connected to both #2 and #3 as parents
  (crossover edge)

Click #3 — the right panel shows the equivalence counterexample
`counterexample: {"args": ["0"], "kwargs": {}}` — proof that `fib(0)`
triggered the rejection.

## Modifying the demo for your own target

The demo is ~200 lines of pure Python. You can fork it and swap out the
target function to sanity-check the pipeline against any callable before
wiring up a real LLM-driven run.

### Change the target function

Edit [`examples/demo_run.py`](../examples/demo_run.py):

```python
VARIANTS: dict[str, str] = {
    "baseline": textwrap.dedent("""
        # your reference implementation here
        def your_fn(x):
            ...
    """).strip(),
    "candidate_a": textwrap.dedent("""
        # an optimisation attempt
        def your_fn(x):
            ...
    """).strip(),
    # ...add as many variants as you want
}
```

And update `BENCH` to import and benchmark `your_fn` instead of `fib`:

```python
BENCH = textwrap.dedent("""
    import json, time
    from your_module import your_fn
    # ...your benchmark here, printing JSON to stdout
""").strip()
```

### Change the metrics

The spec's `metrics` list drives reviewer decisions. Add or remove
entries in `_build_spec()`:

```python
Metric(name="memory_mb", optimise=OptimiseDirection.MINIMIZE),
Metric(name="accuracy",  optimise=OptimiseDirection.MAXIMIZE, minimum=0.9),
```

Make sure your benchmark emits a matching key in its JSON output.

### Change the plan

The `PLAN` list defines round/operator/variant pairings. Add more rounds
or more candidates per round:

```python
PLAN = [
    ("1", "explore",   [],         1, "baseline",       "Reference."),
    ("2", "mutate",    ["1"],      2, "candidate_a",    "Try idea A."),
    ("3", "mutate",    ["1"],      2, "candidate_b",    "Try idea B."),
    ("4", "crossover", ["2", "3"], 3, "candidate_c",    "Combine A and B."),
    ("5", "explore",   [],         3, "candidate_d",    "Different paradigm."),
]
```

Tune `EvolutionSpec(rounds=..., candidates_per_round=...)` in
`_build_spec` to match.

### Change the equivalence strategy

The demo uses `tuples(integers(min_value=0, max_value=20))` for `fib(n)`
inputs. Swap to a strategy matching your function's signature:

```python
# single float argument
strategy = tuples(floats(min_value=-1e6, max_value=1e6, allow_nan=False))

# list argument
strategy = tuples(lists(integers(), max_size=100))

# keyword arguments
strategy = fixed_dictionaries({"x": integers(), "cap": integers()})
```

See [`hypothesis` strategies docs](https://hypothesis.readthedocs.io/en/latest/data.html).

### Swap the mock reviewer for a real one

The demo's `_review()` function is a ~15-line heuristic. To drive a run
with Claude as the reviewer, replace it with a call to `/reviewer`:

```python
# Instead of:
verdict = _review(scored, baseline_us)

# Pseudo-code — ask your agent runner to invoke the reviewer skill:
verdict = agent.invoke("reviewer", candidate_id=cid, spec=spec)
```

At that point you are most of the way to a full Claude-driven run — you
just replace each role (`_review`, the hardcoded `VARIANTS`, the hardcoded
`PLAN`) with agent invocations.
